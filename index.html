Chapter 01 — Foundations

What is Nginx?

Nginx (pronounced "engine-x") is a high-performance, open-source web server that also works as a reverse proxy, load balancer, mail proxy, and HTTP cache. Originally created by Igor Sysoev in 2004, it is now used by millions of websites worldwide.

  • Understand what Nginx is and what it does
  • Learn how Nginx handles concurrency differently from Apache
  • Know when to use Nginx vs other web servers
Core Architecture

Nginx uses an event-driven, asynchronous, non-blocking architecture. Unlike Apache which spawns a new thread or process per connection, Nginx uses a master process with multiple worker processes that each handle thousands of connections simultaneously using the OS event loop.

FeatureNginxApache
ArchitectureEvent-driven asyncProcess/thread per request
Memory usageVery lowHigher under load
Static filesExtremely fastGood
Config styleBlock-basedDirective-based (.htaccess)
Reverse proxyBuilt-in, excellentModule required
What Nginx Can Do
  • Serve static files (HTML, CSS, JS, images) at very high speed
  • Act as a reverse proxy in front of app servers (Node.js, Python, PHP)
  • Load balance traffic across multiple backend servers
  • Terminate SSL/TLS so your app doesn't have to
  • Cache responses to reduce backend load
  • Rate limit and protect APIs from abuse
Nginx is the most commonly used web server for CTF challenges, real-world deployments, and cloud infrastructure. Understanding it is essential for both offensive and defensive security work.

Installation

Installing Nginx is straightforward across all major Linux distributions. This chapter covers installation on Ubuntu/Debian and CentOS/RHEL, plus verifying the install and managing the service.

Ubuntu / Debian / Kali
bash
# Update package list
sudo apt update

# Install Nginx
sudo apt install nginx -y

# Start and enable on boot
sudo systemctl start nginx
sudo systemctl enable nginx

# Check status
sudo systemctl status nginx
CentOS / RHEL / Fedora
bash
# Install Nginx
sudo dnf install nginx -y

# Start and enable
sudo systemctl start nginx
sudo systemctl enable nginx
Verify Installation
bash
# Check version
nginx -v

# Test config file (very useful)
sudo nginx -t

# Check listening ports
ss -tlnp | grep nginx

# Open in browser
curl http://localhost
Service Management Commands
CommandWhat it does
systemctl start nginxStart the service
systemctl stop nginxStop the service
systemctl restart nginxFull restart (brief downtime)
systemctl reload nginxReload config with zero downtime
nginx -tTest config for syntax errors
nginx -s reloadSend reload signal to master process
Always run nginx -t before reloading. A config syntax error will prevent reload and can bring down your site.
Chapter 03 — Foundations

Config Structure

The Nginx configuration file uses a block-based syntax. Understanding the file layout is essential before writing any rules. The main config file is at /etc/nginx/nginx.conf.

File Layout
PathPurpose
/etc/nginx/nginx.confMain config file
/etc/nginx/sites-available/All virtual host configs
/etc/nginx/sites-enabled/Active configs (symlinks)
/etc/nginx/conf.d/Extra configs (auto-included)
/var/www/html/Default web root
/var/log/nginx/Access and error logs
Context Hierarchy
nginx
# Global context (main)
worker_processes auto;
error_log  /var/log/nginx/error.log warn;

# Events context
events {
    worker_connections  1024;
}

# HTTP context
http {
    include       mime.types;
    default_type  application/octet-stream;
    sendfile      on;
    keepalive_timeout 65;

    # Server context lives inside http
    server {
        listen      80;
        server_name example.com;

        # Location context lives inside server
        location / {
            root   /var/www/html;
            index  index.html;
        }
    }
}
Key Directives Explained
DirectiveContextMeaning
worker_processesmainNumber of worker processes (auto = CPU cores)
worker_connectionseventsMax connections per worker
includeanyInclude another config file
listenserverPort and address to listen on
server_nameserverDomain name(s) this block handles
rootserver/locationWeb root directory
indexserver/locationDefault index file
Every directive ends with a semicolon ; — forgetting it causes a syntax error. Blocks use curly braces { } with no semicolon after the closing brace.
Chapter 04 — Core Concepts

Server Blocks

Server blocks (called Virtual Hosts in Apache) allow you to host multiple websites on a single server. Nginx picks which server block to use based on the domain name in the HTTP request.

Basic Static Site
nginx
server {
    listen       80;
    server_name  example.com www.example.com;
    root         /var/www/example;
    index        index.html index.htm;

    location / {
        try_files $uri $uri/ =404;
    }
}
Enabling a Site
bash
# Create config in sites-available
sudo nano /etc/nginx/sites-available/example.com

# Create symlink to enable it
sudo ln -s /etc/nginx/sites-available/example.com /etc/nginx/sites-enabled/

# Test and reload
sudo nginx -t
sudo systemctl reload nginx
Multiple Sites on One Server
nginx
# Site 1 — site-a.com
server {
    listen       80;
    server_name  site-a.com;
    root         /var/www/site-a;
    location / { try_files $uri $uri/ =404; }
}

# Site 2 — site-b.com
server {
    listen       80;
    server_name  site-b.com;
    root         /var/www/site-b;
    location / { try_files $uri $uri/ =404; }
}
Redirect HTTP → HTTPS
nginx
server {
    listen       80;
    server_name  example.com www.example.com;
    return       301 https://$host$request_uri;
}
The try_files directive is key for SPAs (React, Vue). Use try_files $uri $uri/ /index.html to serve the app for all routes instead of returning 404.
Chapter 05 — Core Concepts

Location Blocks

Location blocks define how Nginx handles requests for specific URL paths. They are the most powerful tool in Nginx config and support exact matching, prefix matching, and regex patterns.

Match Modifiers
ModifierTypePriority
= /pathExact matchHighest
^~ /pathPrefix, no regex check afterHigh
~ patternRegex, case-sensitiveMedium
~* patternRegex, case-insensitiveMedium
/pathPrefix match (fallback)Lowest
Examples
nginx
server {
    listen 80;
    server_name example.com;

    # Exact match — only /
    location = / {
        return 200 "Root";
    }

    # Serve static assets with caching
    location ~* \.(css|js|png|jpg|ico|woff2)$ {
        root      /var/www/html;
        expires   30d;
        add_header Cache-Control "public";
    }

    # Block access to hidden files
    location ~ /\. {
        deny all;
    }

    # API prefix — proxy to backend
    location /api/ {
        proxy_pass http://127.0.0.1:3000;
    }

    # Default — serve files
    location / {
        root      /var/www/html;
        try_files $uri $uri/ =404;
    }
}
Named Locations
nginx
location / {
    try_files $uri $uri/ @fallback;
}

# Named location used as fallback
location @fallback {
    proxy_pass http://127.0.0.1:5000;
}
Location block order matters. Nginx picks the most specific match, not the first match. Exact = always wins, followed by ^~, then regex, then prefix.
Chapter 06 — Core Concepts

Reverse Proxy

A reverse proxy sits in front of your application server, forwarding client requests to it and returning the response. Nginx excels at this — hiding your backend, adding SSL, and handling high traffic.

Basic Reverse Proxy
nginx
server {
    listen      80;
    server_name example.com;

    location / {
        proxy_pass         http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header   Host              $host;
        proxy_set_header   X-Real-IP         $remote_addr;
        proxy_set_header   X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header   X-Forwarded-Proto $scheme;
        proxy_set_header   Upgrade           $http_upgrade;
        proxy_set_header   Connection        "upgrade";
    }
}
Reusable Proxy Params

Instead of repeating proxy headers everywhere, put them in a shared file:

bash
# Create /etc/nginx/proxy_params
sudo nano /etc/nginx/proxy_params
nginx — /etc/nginx/proxy_params
proxy_http_version 1.1;
proxy_set_header   Host              $host;
proxy_set_header   X-Real-IP         $remote_addr;
proxy_set_header   X-Forwarded-For   $proxy_add_x_forwarded_for;
proxy_set_header   X-Forwarded-Proto $scheme;
proxy_set_header   Upgrade           $http_upgrade;
proxy_set_header   Connection        "upgrade";
nginx — usage
location / {
    include    /etc/nginx/proxy_params;
    proxy_pass http://127.0.0.1:3000;
}
Proxy Timeouts
nginx
location / {
    proxy_pass            http://127.0.0.1:3000;
    proxy_connect_timeout 60s;
    proxy_send_timeout    60s;
    proxy_read_timeout    60s;
}
The X-Real-IP and X-Forwarded-For headers are critical — without them your app sees Nginx's IP (127.0.0.1) for every request instead of the real client IP.
Chapter 07 — Advanced

Load Balancing

Load balancing distributes incoming traffic across multiple backend servers to prevent overload and increase availability. Nginx has built-in support for several balancing algorithms.

Upstream Block
nginx
# Define upstream group
upstream backend {
    server 10.0.0.1:3000;
    server 10.0.0.2:3000;
    server 10.0.0.3:3000;
}

server {
    listen 80;
    location / {
        proxy_pass http://backend;
    }
}
Balancing Algorithms
nginx
# Round Robin (default) — rotates equally
upstream backend {
    server 10.0.0.1:3000;
    server 10.0.0.2:3000;
}

# Least Connections — sends to server with fewest active
upstream backend {
    least_conn;
    server 10.0.0.1:3000;
    server 10.0.0.2:3000;
}

# IP Hash — same client always hits same server (session sticky)
upstream backend {
    ip_hash;
    server 10.0.0.1:3000;
    server 10.0.0.2:3000;
}

# Weighted — send more traffic to powerful server
upstream backend {
    server 10.0.0.1:3000 weight=3;
    server 10.0.0.2:3000 weight=1;
}
Health Checks & Failover
nginx
upstream backend {
    server 10.0.0.1:3000 max_fails=3 fail_timeout=30s;
    server 10.0.0.2:3000 max_fails=3 fail_timeout=30s;
    # Backup — only used when all above are down
    server 10.0.0.3:3000 backup;
}
Use ip_hash when your app stores sessions in memory. Without it, the same user might hit a different backend on each request and lose their session.
Chapter 08 — Advanced

SSL / TLS

HTTPS is mandatory in modern web deployments. Nginx handles SSL termination — it decrypts HTTPS from clients and forwards plain HTTP to your backend. The easiest way to get a free cert is Certbot with Let's Encrypt.

Install Certbot (Free SSL)
bash
# Install Certbot
sudo apt install certbot python3-certbot-nginx -y

# Get cert (auto-configures Nginx)
sudo certbot --nginx -d example.com -d www.example.com

# Test auto-renewal
sudo certbot renew --dry-run
Manual SSL Config
nginx
# Redirect HTTP to HTTPS
server {
    listen      80;
    server_name example.com www.example.com;
    return      301 https://$host$request_uri;
}

# HTTPS server block
server {
    listen              443 ssl http2;
    server_name         example.com www.example.com;

    ssl_certificate     /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

    # Modern SSL settings
    ssl_protocols       TLSv1.2 TLSv1.3;
    ssl_ciphers         HIGH:!aNULL:!MD5;
    ssl_prefer_server_ciphers on;
    ssl_session_cache   shared:SSL:10m;
    ssl_session_timeout 10m;

    location / {
        root      /var/www/html;
        try_files $uri $uri/ =404;
    }
}
HSTS Header
nginx
# Force HTTPS for 1 year via browser header
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
Only add HSTS after confirming HTTPS works perfectly. Once a browser receives this header, it will refuse to connect over HTTP for the specified duration.
Chapter 09 — Advanced

Security Hardening

Out of the box, Nginx leaks version information, allows all HTTP methods, and has no rate limiting. These steps harden it against common attacks and information disclosure.

Hide Nginx Version
nginx
# In http context — stops "nginx/1.18.0" in Server header
server_tokens off;
Security Headers
nginx
server {
    # Prevent clickjacking
    add_header X-Frame-Options          "SAMEORIGIN"   always;
    # Block MIME sniffing
    add_header X-Content-Type-Options   "nosniff"      always;
    # Enable browser XSS filter
    add_header X-XSS-Protection         "1; mode=block" always;
    # Referrer control
    add_header Referrer-Policy           "strict-origin-when-cross-origin" always;
    # Content Security Policy
    add_header Content-Security-Policy  "default-src 'self'" always;
}
Rate Limiting
nginx
# Define rate limit zone in http context
limit_req_zone $binary_remote_addr zone=api:10m rate=10r/s;

server {
    location /api/ {
        # Allow burst of 20, queue the rest
        limit_req zone=api burst=20 nodelay;
        proxy_pass http://127.0.0.1:3000;
    }
}
Block Bad Requests
nginx
# Block access to hidden files (.env, .git, etc.)
location ~ /\. {
    deny all;
    access_log off;
    log_not_found off;
}

# Block bad user agents
if ($http_user_agent ~* (wget|curl|nikto|sqlmap|nmap)) {
    return 403;
}

# Limit request methods
if ($request_method !~ ^(GET|POST|HEAD)$) {
    return 405;
}
IP Allowlist / Denylist
nginx
# Block a specific IP
location / {
    deny  192.168.1.100;
    allow all;
}

# Allow only specific IPs (admin panel)
location /admin {
    allow 10.0.0.0/24;
    deny  all;
}
Chapter 10 — Advanced

Logging & Performance

Nginx has powerful logging capabilities and many performance tuning options. This final chapter covers reading logs, customizing log formats, enabling compression, caching, and tuning worker settings.

Log Files
bash
# Access log — every request
tail -f /var/log/nginx/access.log

# Error log — errors and warnings
tail -f /var/log/nginx/error.log

# Watch both simultaneously
tail -f /var/log/nginx/access.log /var/log/nginx/error.log
Custom Log Format
nginx
# In http context — define custom format
log_format detailed '$remote_addr - $remote_user [$time_local] '
                     '"$request" $status $body_bytes_sent '
                     '"$http_referer" "$http_user_agent" '
                     'rt=$request_time uct=$upstream_connect_time';

server {
    access_log /var/log/nginx/access.log detailed;
}
Gzip Compression
nginx
# In http context
gzip              on;
gzip_vary         on;
gzip_min_length   1024;
gzip_comp_level   6;
gzip_types        text/plain text/css application/json
                   application/javascript text/xml
                   application/xml image/svg+xml;
Worker Tuning
nginx
# In main context
worker_processes     auto;       # One per CPU core
worker_rlimit_nofile 65535;     # Max open files per worker

events {
    worker_connections 4096;     # Connections per worker
    use                epoll;     # Linux kernel event model
    multi_accept       on;        # Accept multiple connections at once
}

http {
    sendfile            on;        # Efficient file sending
    tcp_nopush          on;        # Batch send packets
    tcp_nodelay         on;        # No delay on small packets
    keepalive_timeout   65;
    keepalive_requests  1000;
    client_max_body_size 16M;     # Limit upload size
}
Static File Caching
nginx
location ~* \.(css|js|png|jpg|jpeg|gif|ico|svg|woff|woff2)$ {
    expires    1y;
    add_header Cache-Control "public, immutable";
    access_log off;
}
Max concurrent connections = worker_processes × worker_connections. With 4 cores and 4096 connections each, you can handle ~16,000 simultaneous connections.
